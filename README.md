Тема:
Система контролю версій Git та співпраця через GitHub

Опис:
Git — це розподілена система контролю версій (VCS, Version Control System), створена Лінусом Торвальдсом у 2005 році для розробки ядра Linux. Сьогодні Git є де-факто стандартом для контролю версій у програмному забезпеченні та використовується мільйонами розробників по всьому світу.
Система контролю версій дозволяє відстежувати зміни в коді з часом, повертатися до попередніх версій, працювати над проектом командою без конфліктів, експериментувати з новими функціями в ізольованих гілках, та багато іншого. Git зберігає повну історію проекту, що дозволяє відповісти на питання "хто, що, коли і чому змінив" у будь-який момент часу.
Розуміння Git є критично важливим для сучасних розробників, незалежно від спеціалізації. Це не просто інструмент для збереження коду — це фундамент для ефективної командної роботи, continuous integration/deployment, code review процесів та професійної розробки ПЗ.

Мета:

Працювати з .gitignore та захищати конфіденційні дані

Навички для майбутньої кар'єри:

Версіонування коду в командній розробці
Code review та collaboration workflows

Portfolio management через GitHub

Контрольні питання з Git:

1. Основи Git
Яка різниця між Git та GitHub?
- Це зовсім різні речі. Перше це система контролю версій, а друге це онлайн сервіс для зберігання репозиторіїв.
Що означає "розподілена" система контролю версій?
- Всі дані репозиторію користувачі можуть мати локально.
Яка різниця між git init та git clone?
- Ініт створює репозиторій, а клон копіює вже існуючий на комп'ютер.

2. Staging Area та Commits
Що таке Staging Area і навіщо вона потрібна?
- Область де фвйли знаходяться після git add.
Яка різниця між git add . та git add -A?
- . додає змінені файли в папці, -А додає взагалі все, навіть видалені файли.
Як написати гарне commit message?
- Згідно з теоретичним матеріалом Conventional Commits — стандарт для повідомлень: <type>[optional scope]: <description>.
Як виправити останній коміт?
- Написати команду ще раз додавши --amend.

3. Гілки та злиття
Що таке гілка в Git і чому вони "легковажні"?
- Набір комітів або вказівник на коміт для паралельної роботи.
Яка різниця між git merge та git rebase?
- merge об'єднує історію гілок, і має власний merge-коміт, а rebase переміщює коміти так, щоб історія була лінійною.
Що таке fast-forward merge?
- Злиття без коміту.
Коли виникають merge конфлікти і як їх уникнути?
- Коли є зміни в одних і тих же рядках та вони різні, а уникнути можна часто роблячі pull та обходячи довготривалі гілки.
Чому не можна робити rebase публічних комітів?
- Це змінює історію комітів, така собі ідея.

4. Віддалені репозиторії
Що таке origin?
- Стандартна назва віддаленого репозиторію.
Яка різниця між git fetch та git pull?
- fentch просто завантажує зміни з віддаленого репозиторію, а pull це fetch та merge одночасно.
Що робить git push --force і чому це небезпечно?
- Перезаписує віддалену гілку не дивлячись ні на що, це може стерти щось важливе і завадити роботі інших.
Як видалити віддалену гілку?
- За допомогою git push origin --d.

5. Скасування змін
Яка різниця між git reset --soft, --mixed та --hard?
- Перше залишає зміни в staging area, друге залишає у файлах і третє все скидає.
Яка різниця між git reset та git revert?
- git reset змінює історію локально і може видаляти коміти, а git revert створює новий коміт, який відміняє зміни.
Як скасувати git add (видалити файл зі staging)?
- За допомогою git restore --staged.
Як повернути видалений файл?
- git restore.

6. Безпека
Що не повинно потрапляти в Git репозиторій?
- Паролі, великі та непотрібні файли.
Що робити якщо випадково закомітили пароль?
- git reset.
HTTPS vs SSH для GitHub — що краще?
- SHH безпечніше та зручніше, бо в HTTPS треба постійно вводити пароль.
Що таке Personal Access Token і навіщо він потрібен?
- Токен, якій замінює пароль для HTTPS, потрібен для безпечного пушу.

7. GitHub
Що таке Pull Request?
- Запит на злиття змін з однієї гілки в іншу.
Яка різниця між Issues та Pull Requests?
- Перше це проблеми та обговорення, друге це пропозиція.
Що таке Fork і навіщо він потрібен?
- Копія чужого репозиторію, щоб щось там зробити і запропонувати Pull Request.
Що таке GitHub Actions?
- Інструмент для автоматизації різних процесів.

8. Best Practices
Як часто треба робити коміти?
- Чим частіше, тим краще, але все має бути логічно.
Скільки часу має жити feature branch?
- До завершення роботи над нею.
Як організувати git workflow в команді?
- feature-branches, Pull Requests, ну і всі інші пункти розглянуті раніше.
Чи треба комітити node_modules або інші залежності?
- Ні, додаються в .gitignore.
